# AIDeliveryOS Code of Conduct

## Purpose

AIDeliveryOS is an open-source project focused on Context Engineering, implementation readiness, AI-assisted development, and project delivery systems.

This Code of Conduct exists to create a professional, collaborative, and respectful environment for all contributors.

## Community Values

The AIDeliveryOS community is built around:

- Intellectual honesty
- Continuous learning
- Clear communication
- Evidence-based decisions
- Respectful disagreement
- Practical problem solving

These values guide how contributors collaborate, review ideas, and improve the framework.

## Our Principles

We expect contributors to:

- Be respectful and professional.
- Focus on ideas, evidence, and constructive discussion.
- Assume positive intent.
- Welcome different perspectives and experiences.
- Provide feedback that helps improve the project.
- Support a learning-oriented community.

## Expected Behavior

Examples of positive behavior include:

- Sharing knowledge and experience.
- Providing thoughtful feedback.
- Helping others learn the framework.
- Reporting issues respectfully.
- Contributing improvements to documentation, examples, templates, and methodology.
- Engaging in evidence-based discussions.

## Unacceptable Behavior

Examples of unacceptable behavior include:

- Harassment, discrimination, or personal attacks.
- Trolling or intentionally disruptive behavior.
- Abusive, threatening, or insulting language.
- Publishing private or confidential information.
- Deliberately misleading contributors.
- Spam, promotion, or unrelated content.

## Decision-Making Philosophy

The project encourages healthy debate and critical thinking.

Strong ideas should be challenged.

Assumptions should be questioned.

Decisions should be supported by evidence, tradeoffs, and documented reasoning.

AIDeliveryOS is built on several principles:

- Context before code.
- Validation before complexity.
- Discovery before architecture.
- Evidence before opinion.
- Reusable systems over one-off solutions.

Disagreements are expected.

When disagreements occur, contributors should focus on evidence, tradeoffs, and project goals.

## Contribution Standards

Contributors should strive to:

- Improve clarity.
- Reduce ambiguity.
- Document assumptions.
- Surface risks.
- Improve implementation readiness.
- Keep documentation consistent.

### High-Value Contributions

Examples of valuable contributions include:

- Improving methodology documentation
- Expanding Project Context Package examples
- Identifying gaps in implementation readiness
- Improving templates and frameworks
- Clarifying architecture and technology guidance
- Sharing lessons learned from real-world usage

## Reporting Issues

If you experience behavior that violates this Code of Conduct, please open an issue or contact the project maintainers.

All reports will be reviewed fairly and respectfully.

## Enforcement

Project maintainers reserve the right to remove, edit, or reject contributions that do not align with this Code of Conduct.

Repeated violations may result in removal from project participation.

### Enforcement Philosophy

The goal of enforcement is not punishment.

The goal is to maintain a productive environment focused on learning, collaboration, and improving project outcomes.

Whenever possible, maintainers should seek clarification and resolution before taking corrective action.

## Scope

This Code of Conduct applies to:

- GitHub discussions
- Issues
- Pull requests
- Documentation contributions
- Community interactions related to the project

## Community Goal

AIDeliveryOS exists to help teams make better decisions before implementation begins.

Contributors are encouraged to improve the framework, challenge assumptions, and share practical experience that helps others deliver better outcomes.

## Final Principle

AIDeliveryOS is built on the belief that many project failures occur long before implementation starts.

The objective is to improve project outcomes through better context, clearer decisions, visible assumptions, and implementation readiness.

Context before code.
