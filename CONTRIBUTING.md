# Contributing to AIDeliveryOS

Thank you for your interest in contributing to AIDeliveryOS.

This project is focused on Context Engineering, implementation readiness, AI-assisted development, and project delivery systems.

The goal is to help teams make better decisions before implementation begins.

## Contribution Philosophy

AIDeliveryOS is a methodology-first project.

The objective is not to build features as quickly as possible.

The objective is to improve implementation outcomes through better context, better decisions, and better project preparation.

Contributions should prioritize clarity, usefulness, and practical application.

## Before You Contribute

Please review:

- README.md
- CODE_OF_CONDUCT.md
- docs/methodology.md
- docs/system-overview.md
- templates/project-context-package-template.md

Understanding the methodology before contributing will help maintain consistency across the project.

## What We Are Looking For

High-value contributions include:

- Improvements to methodology documentation
- Additional Project Context Package examples
- Better implementation readiness frameworks
- Architecture and technology guidance improvements
- New templates and starter kits
- Corrections, clarifications, and documentation improvements
- Real-world lessons learned and case studies

## Contribution Principles

All contributions should support the project's core principles:

- Context before code
- Validation before complexity
- Discovery before architecture
- Evidence before opinion
- Reusable systems over one-off solutions

### Preferred Contribution Characteristics

Strong contributions are:

- Practical
- Reusable
- Evidence-based
- Easy to understand
- Easy to adopt
- Aligned with implementation readiness

When proposing changes, explain:

- The problem being solved
- The reasoning behind the change
- Expected benefits
- Potential tradeoffs

## Types of Contributions

### Documentation

Documentation improvements are encouraged and often provide the highest value.

Examples:

- Clarifying concepts
- Fixing inaccuracies
- Improving examples
- Expanding explanations

### Examples

Example Project Context Packages should be realistic, practical, and implementation-focused.

Examples should document:

- Discovery findings
- Stakeholder alignment
- Assumptions
- Risks
- Recommendations
- Success metrics
- Readiness assessments

### Framework Improvements

Framework updates should:

- Improve decision quality
- Reduce ambiguity
- Increase implementation readiness
- Remain simple and practical

## Documentation Quality Standards

Documentation should:

- Use clear language.
- Define assumptions.
- Explain tradeoffs.
- Separate facts from opinions.
- Focus on implementation readiness.
- Remain actionable and practical.

## Pull Request Guidelines

Before submitting a pull request:

1. Review existing documentation.
2. Ensure your contribution aligns with project principles.
3. Document assumptions and rationale.
4. Verify consistency with existing terminology.
5. Keep changes focused and easy to review.

### Pull Request Template

Include:

- Problem Statement
- Proposed Change
- Expected Outcome
- Tradeoffs
- Related Documentation

## Decision-Making Standards

Contributions are evaluated based on:

| Criteria | Question |
|---|---|
| Clarity | Does this reduce ambiguity? |
| Practicality | Can teams use this immediately? |
| Consistency | Does it align with the methodology? |
| Evidence | Is the recommendation supported? |
| Reusability | Can others benefit from it? |

### Decision Hierarchy

When evaluating competing ideas, prioritize:

1. Evidence
2. User value
3. Simplicity
4. Reusability
5. Complexity

Complex solutions should require stronger justification than simple solutions.

The objective of a contribution is not to add more content.

The objective is to improve the quality of decisions that occur before implementation begins.

Every contribution should help teams create clearer requirements, stronger plans, better architectural decisions, and improved implementation outcomes.

Context before code.
